# MetroAI v0.7.0 Roadmap (재정렬 v2)

> 최초 작성: 2026-05-18
> **재정렬: 2026-05-19** — 가상 손님 시나리오 (SEM lab 운영자) 점검 결과 v0.6.0 fit 45% 확인.
> 외부 노출 (P1) 보다 **손님 시나리오 cover (P0 신설)** 가 더 큰 leverage 라고 재판단.

---

## 0. 재정렬 배경 — 5/19 가상 손님 시나리오 점검

사용자 (김용범) 가 가상 SEM lab 운영자 입장에서 MetroAI 웹앱 사용 시나리오 7단계 가정:

1. SEM 장비 KOLAS 인정 받고 싶어서 웹앱 도착
2. 분야별 기본 가이드 + 신청서 양식 다운로드
3. 기본 정보 입력하면 신청서·SOP·QC 절차서 자동 생성
4. SEM 분야 KOLAS 심사 절차 상세 안내
5. 우리 lab SOP 업로드 → 자동 갭 분석
6. "이렇게 바꾸면 위험 X% 감소" what-if 시뮬레이션
7. 신청 → 평가 동행 → 사후 서비스까지 end-to-end

**v0.6.0 점검 결과:**

| 단계 | fit | 한계 |
|---|---|---|
| 1 진입 | 50% ⚠️ | 일반 KOLAS 메시징. SEM 특화 진입로 없음 |
| 2 가이드/템플릿 | 30% ⚠️ | 측정 불확도 템플릿은 있으나 인정 가이드 콘텐츠 부재 |
| 3 자동 작성 | 25% ❌ | KAB-F-21-2026 자동 채움 없음 |
| 4 분야별 심사 안내 | 20% ❌ | 분야별 콘텐츠 부재 |
| 5 SOP 비교 | 50% ⚠️ | SOP 갭 페이지는 있으나 PDF 파싱 stub |
| 6 시뮬레이션 | 70% ✅ | 감사 위험 페이지 (v2 block 3) cover. 단, 권고는 generic |
| 7 end-to-end | 10% ❌ | 컨설팅 영역. consulting layer 부재 |

**전체 fit: ~45%**.

**판단:** 사용자 1명 확보 (P1 outbound) 보다 **fit 45% → 80% 끌어올리는 P0 구현** 이 같은 시간 대비 더 큰 leverage. 첫 사용자가 와도 fit 45% 면 churn 이 빠릅니다.

## 1. v0.7.0 우선순위 (재정렬, 5/19 → 8/19)

### Priority 0 (신설, 6/19 까지) — 손님 시나리오 cover (fit 45% → 80%)

| 항목 | 시간 | 산출물 | 점검 시나리오 |
|---|---|---|---|
| **P0-1** 분야별 진입 wizard | 4일 | 첫 페이지에서 "SEM / AFM / TEM / OCD / 일반" 선택 → 분야별 dashboard 진입 | #1 진입 |
| **P0-2** 분야별 KOLAS 가이드 콘텐츠 | 1주 | SEM/AFM/TEM/OCD 4종 인정 가이드 (KOLAS-G-002 발췌 + 분야 commentary + 흔한 부적합) | #2, #4 |
| **P0-3** KOLAS 신청서 자동 작성기 | 1주 | 조직 프로필 1회 입력 → KAB-F-21-2026 양식 자동 채움 → PDF 출력 | #3 |
| **P0-4** 실 SOP 업로드 → 자동 갭 분석 | 1주 | PDF/Word/Excel 파싱 → 분야별 rule-base + LLM 으로 갭 리스트 + 권고 | #5 |

**목표 (6/19):** 가상 손님 시나리오 7단계 점검 시 fit 80%+ (단계 1·2·3·4·5 모두 ✅).

### Priority 1 (강력 권장, 7/19 까지) — 실 데이터 확보 (기존 P2)

| 항목 | 시간 | 산출물 |
|---|---|---|
| **P1-1** KOLAS 공개 부적합 통계 수집 (knab.go.kr) | 1주 | n ≥ 100 합성 대비 진짜 |
| **P1-2** 협력 기관 1곳 데이터 동의 | 2주 | 익명화 n ≥ 50 |
| **P1-3** GBT 모델 실데이터 retrain | 1주 | 정직 acc 외부 공개 |

**목표 (7/19):** kolas-audit-predictor 실데이터 acc 외부 공개 + 손님 시나리오 #6 시뮬레이션 정확도 60.6% synthetic → 65%+ 실데이터.

### Priority 2 (가능하면, 8/19 까지) — 외부 신호 + 인프라

기존 P1 (outbound) + P3 (기능 깊이) 통합. 손님 시나리오 cover 후 outbound 하면 churn 가능성 ↓.

| 항목 | 시간 | 산출물 |
|---|---|---|
| **P2-1** Streamlit Cloud → HF Spaces 이전 | 2일 | sleep 해결 (`docs/HF_SPACES_MIGRATION.md` 가이드 존재) |
| **P2-2** KOLAS 인정 기관 cold email (P0 완성 후 발송) | 1주 | fit 80% 데모 첨부 → 응답 가능성 ↑ |
| **P2-3** Reddit r/MachineLearning + Show HN 게시 | 4시간 | 사용자 직접 (`docs/PROMO_5_19.md` 드래프트 준비됨) |
| **P2-4** LLM-assisted kolas-monitor (실제 추론 layer) | 1주 | 명확한 AI 차별화 |
| **P2-5** demo 영상 3분 (P0 완성 후) | 1일 | YouTube + LinkedIn |

**목표 (8/19):** 외부 사용자 5명+ 응답, GitHub Stars 10+, MCPize subscribers 3+.

### Priority 3 (Optional, 사용자 손) — end-to-end consulting (시나리오 #7)

손님 시나리오 #7 (신청 → 평가 동행 → 사후 서비스) 은 본질적으로 컨설팅 + 인간 lab 매니저 영역. 자동화 적용 어려움. **단, MetroAI 가 보조 도구로 컨설턴트의 작업을 가속화**.

| 항목 | 시간 | 산출물 |
|---|---|---|
| **P3-1** 컨설팅 SOP 자료집 작성 (사용자 본인 노하우) | 1주 | KOLAS 인정 신청부터 사후심사까지 step-by-step |
| **P3-2** "MetroAI + 인간 컨설팅" 패키지 가격 책정 | 2일 | $X/month MetroAI + $Y/case 컨설팅 |

## 3. 재정렬 이후의 우선순위 logic

**v0.6 의 학습:** 빌드만 잘하고 outbound 안 하면 사용자 0명.
**v0.7 의 학습 (5/19 재정렬):** **fit 45% 인 채로 outbound 해도 churn 가능성 높음.** 사용자 시나리오 cover 가 우선.

추천 순서:
1. **P0 4가지** 먼저 구현 (1개월) — fit 45% → 80%
2. fit 80% 도달하면 **P2 outbound** 시작 — 데모 demo 가 강력해진 상태
3. **P1 실데이터** 는 P2 응답자 중 협력 가능자 발굴
4. **P3 consulting** 은 사용자 본인 시간 가능 시
